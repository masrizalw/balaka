## Highlights

Feature release. Two themes:

- **DRAFT-by-default posting model.** Every high-level feature that previously auto-posted to the general ledger (invoice send, vendor bill approve, fixed-asset acquisition, monthly depreciation, inventory in/out, BOM production completion) now composes journal entries as **DRAFT** through the template engine. Accounting staff reviews and explicitly **POSTs** before the GL changes. Separation of duties between operational entry and ledger control.
- **Search-combobox UX across every unbounded picker.** All `<select>` elements that listed clients, vendors, products, or accounts have been replaced with typeahead search comboboxes that return at most 10 results per query. Forms stay usable as master data grows past a few dozen entries.

## What's New

### DRAFT-by-default rearchitecture

- New `DocumentPostingService` reusable bridge composes a DRAFT transaction from a journal template + slot mappings + variables, with optional `hintTo` account resolution.
- Generic `source_document_type` / `source_document_id` link on `transactions` so a DRAFT can be traced back to its originating invoice, bill, asset, or production order.
- R2 grouping pattern on multi-line documents: one DRAFT per distinct revenue/expense account (lines sharing the same account are merged).
- `FixedAsset.autoPost` opt-in flag ‚Äî the monthly depreciation scheduler honors it instead of always posting; default is PENDING for accounting review.
- Inventory transactions (PURCHASE, SALE, PRODUCTION_IN/OUT, ADJUSTMENT) no longer auto-post; same DRAFT bridge.
- Free-form journal entry API: `POST /api/transactions/journal-entry` for arbitrary debit/credit lines without a template (closing / adjusting / opening entries).
- Fiscal-year closing entries are an exception ‚Äî they remain POSTED inline (explicit user action, period-review verifies).

### Posting bridge accounts on company configuration

- `company_config` gets four new account-id slots: receivable (AR), payable (AP), output tax (PPN Keluaran), input tax (PPN Masukan). FKs to `chart_of_accounts`.
- Settings UI section "Akun Posting (Jurnal Otomatis)" with four account pickers.
- Industry seed data populates the slots automatically; tenants upgrading from 2026.05 must fill them before the next invoice send / bill approve.

### Search-combobox UX

Server search endpoints, top-10 results each:
- `GET /clients/search` ‚Äî active clients by code or name
- `GET /vendors/search` ‚Äî active vendors by code or name

Existing `GET /products/search` and `GET /accounts/search` (added in earlier patches) cover the other two.

Three Alpine components in `alpine-components.js` (CSP-safe, no inline expressions):
- `clientPicker` ‚Äî invoice form, project form, invoices/projects list filters
- `vendorPicker` ‚Äî bill form, bills list filter
- `accountPicker` ‚Äî settings/company (4 slots), asset form (fundingAccount), bill per-line expense account, transaction per-line account, quick-transaction per-line account, journal entry per-line account

Per-line pickers inside `x-for` use embedded picker state owned by the parent Alpine component (per-line `label / selectedId / results / pickerOpen`) ‚Äî `x-model` on inputs inside `x-for` proved unreliable for two-way binding under late-init, so amount/credit inputs use `:value` + `@input` handlers reading `$event.target.value` explicitly.

Demo loaders (`DemoDataLoaderBase`) and functional page objects (`TransactionFormPage`) rewritten to drive comboboxes ‚Äî type a code prefix, wait for debounce, click the matching `[data-testid='*-picker-result']`.

### Invoice/bill product picker

- Invoice lines reference `product_id`; the bound product's `sales_account` drives the revenue side of the DRAFT.
- Bill lines reference `expense_account_id` per line (combobox).
- DRAFT generation groups lines by revenue/expense account so multi-product invoices and multi-category bills produce one DRAFT per account rather than one per line.

### Demo data

- Demo loaders now populate vendors (`28_vendors.csv`) and bills (`29_bills.csv`) across all four industry instances (it-service, online-seller, campus, coffee-shop).
- Invoices added to it-service / online-seller / campus (coffee-shop stays cash-only as before).
- Asset funding accounts seeded; asset auto-post checkbox state seeded.

## Bug Fixes

- Jackson + Hibernate-proxy serialization: `@JsonIgnoreProperties({"hibernateLazyInitializer","handler"})` on `Product` and `ChartOfAccount`. Previously caused intermittent invoice-line serialization failures when a test class was run alongside siblings (cache state interaction).
- `SettingsControllerTest.shouldRejectInvalidCompanySettings`: `CompanyConfigForm` was missing the four label fields needed for POST-error re-render against the new picker template; added receivable/payable/outputTax/inputTax label fields.
- `RecurringSchedulerTest.shouldSkipWeekend`: clamps `dayOfMonth` to the entity's `@Max(28)` constraint so the date-dependent test stays valid past the 28th.
- `InvoiceLifecycleTest`: lifecycle bill missing expense account on its single line ‚Äî now sets a 5.1.20 expense account so the new DRAFT bridge succeeds.
- `PaymentTrackingTest` / `StatementTest`: programmatically create a `Product` and bind invoice lines to it before send; the new revenue-recognition bridge requires every line to have a product.
- `FixedAssetServiceTest`, `FixedAssetSchedulerTest`, `FixedAssetPersistenceTest`, `FixedAssetControllerFunctionalTest`: set `fundingAccount` on all fixtures.

## Breaking Changes

**Tenants upgrading from 2026.05 must configure the four posting-bridge accounts on `/settings/company` before the next invoice send / bill approve.** Without them, the DRAFT bridge throws with a clear message ("Akun ‚Ä¶ belum dikonfigurasi") and the operation fails atomically ‚Äî no partial state. Industry seed data already includes the accounts; the gap is only for tenants that imported seed before this release.

**Invoice lines now require a product.** Existing data is fine (lines created before the upgrade keep their NULL product), but new invoice lines created after the upgrade must reference a product. The combobox enforces this on the form.

**Asset acquisition now requires `fundingAccount`.** A bank/AP credit account must be chosen on the form. Existing assets are unaffected (legacy assets show NULL `id_funding_account`); only new assets are gated.

Otherwise no breaking changes: GL posting still goes through the same template engine; APIs are backward compatible; existing report endpoints unchanged.

## Migration Guide

For a tenant already running 2026.05-RELEASE:

1. Standard deployment (build, copy JAR, restart service). Flyway applies `V002__core_schema.sql` updates idempotently ‚Äî no manual schema work needed; the new `id_*_account` columns on `company_config` are nullable and the new `auto_post` column on `fixed_assets` defaults to `FALSE`.
2. Log in as admin and open **Pengaturan ‚Üí Profil Perusahaan**. In the "Akun Posting (Jurnal Otomatis)" section, pick the four accounts (Piutang, Hutang, PPN Keluaran, PPN Masukan).
3. Optionally edit existing fixed assets to set a `fundingAccount` if you anticipate disposing or otherwise touching their journal entries; legacy assets without one still depreciate.
4. Existing invoice lines that lack a `product_id` keep working (read-only views). New invoice lines created via the form require a product ‚Äî set up the product catalogue (Master Data ‚Üí Produk) before issuing new invoices.

User manual updated to document the new DRAFT workflow, combobox UX, and posting-bridge configuration: see [DRAFT-by-default](https://artivisi.com/balaka/faktur-tagihan.html#draft-by-default) and [Akun Posting Jurnal Otomatis](https://artivisi.com/balaka/setup-awal.html#akun-posting).
